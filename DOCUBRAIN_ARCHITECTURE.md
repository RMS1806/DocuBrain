# DocuBrain: Full-Stack Architecture & Data Flow Guide

This document is designed to give you a complete, top-down understanding of the DocuBrain platform. If you are asked in an interview how your application is built, how the infrastructure manages heavy AI workloads, or how data flows across the system, use this guide.

---

## 🏗️ 1. High-Level Technology Stack
When asked **"What is your stack?"**, here is your answer:

- **Frontend:** React + Vite, styled with custom CSS and Framer Motion for dynamic animations. Deployed on **Vercel**.
- **Backend API:** **FastAPI** (Python). Chosen because of its incredible async performance and automatic OpenAPI generation. Deployed on **Render** (Free Tier).
- **Database:** **PostgreSQL** hosted on **Supabase**, interacting via SQLAlchemy & Asyncpg.
- **Message Broker & Cache:** **Redis** (Managed Cloud).
- **Background Worker:** **Celery**. Required to decouple heavy AI text-processing from the fast web server.
- **AI Vector Engine (RAG):** **ChromaDB** running persistently in-memory to store embeddings, integrated with the **Google Gemini API** for LLM generation and Text Embeddings.

## 🌊 2. Core Workflows & Data Flows
Here are the exact pathways data takes through the system during primary user actions.

### Flow A: The Document Upload & AI AI-Embedding Flow 
*(This is the most complex flow and shows your advanced engineering skills!)*

1. **Upload Request:** The React frontend sends a binary PDF file via `multipart/form-data` to the FastAPI backend (`POST /upload/`).
2. **Local Storage & DB Pending:** FastAPI securely saves the PDF to an ephemeral local disk (`/uploads`) to prevent RAM bloat. It inserts a record into PostgreSQL with `status="pending"`. 
3. **Task Handoff (Redis):** Because processing AI takes time (10-30 seconds), if the API waited, the browser would timeout. Instead, FastAPI sends a message to **Redis** (acting as a Message Broker) and immediately returns a `200 OK` to the frontend.
4. **Celery Worker Execution:**
   - The Celery Background Worker (running simultaneously alongside FastAPI) constantly listens to Redis. It grabs the task.
   - It reads the PDF from the local disk and extracts text using `PyPDF`.
   - It sends the text to the **Gemini Embeddings API** to convert the text into highly dense mathematical arrays (Vectors). *Note: You did this to avoid loading massive HuggingFace models into your limited 512MB RAM!*
   - It stores these vectors into **ChromaDB**.
5. **Completion:** Celery updates the PostgreSQL document status to `"completed"`. When the React frontend periodically refreshes, it sees the new status and updates the UI UI.

### Flow B: The Neural Chat (RAG) Flow
*(RAG = Retrieval-Augmented Generation. This explains how the AI actually "reads" the PDFs).*

1. **User Query:** The user types a question in the React UI (e.g., "What does the contract say about refunds?").
2. **Query Vectorization:** FastAPI receives the question and asks the Gemini Embeddings API to convert the *question* into a Vector.
3. **Similarity Search:** FastAPI takes the question-vector and queries **ChromaDB**. ChromaDB mathematically compares the question against all the PDF text-chunks it has stored, and returns the top 5 most relevant paragraphs.
4. **LLM Prompt Injection:** FastAPI takes those 5 paragraphs and stitches them together with a secret system prompt indicating: _"You are DocuBrain. Answer the user's question using ONLY the provided document context..."_
5. **Real-time Streaming:** FastAPI sends this massive prompt to the **Gemini LLM API**. As Gemini types out the answer word-by-word, FastAPI uses **Server-Sent Events (SSE)** to stream the response chunks back to the React UI in real-time, giving that satisfying "typing" effect.

### Flow C: Fast Data Caching
1. **The Problem:** Querying Supabase PostgreSQL repeatedly for things like the dashboard's "Document List" or "Chat Sessions" causes network bottlenecks.
2. **The Solution:** FastAPI implements an internal **Redis Cache**. When a user requests their documents, FastAPI pulls it from PostgreSQL, but then saves it in Redis for 30 seconds. If they refresh the page within 30 seconds, FastAPI returns the data instantly from Redis, completely bypassing the database!

## 🚀 3. Overcoming Cloud Production Constraints
*Interviewers love hearing about problems you solved. Here is exactly how you engineered around cloud limits:*

**1. The "Split Filesystem" Problem**
In modern PaaS (Platform as a Service) like Render, services are isolated. If FastAPI ran on one machine and Celery ran on another, Celery wouldn't be able to find the PDF files generated by FastAPI!
**Your Solution:** You created a custom `entrypoint.sh` bash script in Docker. When the container starts, it deliberately runs both `celery worker &` and `uvicorn` in the exact same container, guaranteeing they share the same physical ephemeral disk.

**2. The Out of Memory (OOM) 512MB Crash**
Celery natively tries to spawn as many child processes as there are physical CPU cores. In a cloud environment, this triggered 8 worker processes simultaneously, which instantly blew past the 512MB RAM limit on the Free Tier and crashed the server.
**Your Solution:** You surgically constrained the architecture by passing `--concurrency=1` to Celery and `--workers 1` to FastAPI. This ensures strict, tightly controlled memory tracking that fits perfectly under the 512MB ceiling.

**3. The Redis Strict TLS Crash**
Because you hooked into a managed cloud Redis provider (Upstash/Aiven), the database required an encrypted TLS connection (`rediss://`). However, Celery's strict source code immediately crashes on TLS URLs unless mutual certificates are disabled.
**Your Solution:** You wrote a dynamic script wrapper that automatically intercepts `rediss://` environments and appends the `?ssl_cert_reqs=CERT_NONE` URL flag, tricking Celery into successfully creating the handshake without crashing.

## 💡 4. Potential Future Improvements (Interview Ideas)
If asked "What would you improve?":
- **WebHooks/WebSockets over Polling:** Currently, the frontend dashboard uses `setTimeout` and polling to check if an upload is "completed". I plan to implement full zero-latency WebSockets or Supabase Realtime subscriptions so the UI updates instantly when Celery finishes.
- **Dedicated S3/AWS Object Storage:** Currently, PDFs are saved to the ephemeral local disk. If the server scales to multiple horizontal instances, local disks don't sync. I plan to introduce an AWS S3 or MinIO bucket constraint so files are persistent.
